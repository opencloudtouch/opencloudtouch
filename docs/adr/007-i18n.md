# ADR 007 — Internationalization (i18n) with react-i18next

**Date**: 2026-05-01  
**Status**: Accepted  
**Deciders**: OpenCloudTouch maintainers

---

## Context

All user-visible strings in the OpenCloudTouch frontend were hardcoded in German inside JSX
components. This made the application inaccessible to non-German speakers and violated the
project goal of being usable by the global Bose SoundTouch community. With the Bose cloud
shutdown on May 6, 2026, users worldwide need to operate their devices independently — a
German-only UI is a significant barrier.

Additionally, there was no mechanism to persist a user's language preference or to detect the
browser's preferred language on first visit.

---

## Decision

Adopt **react-i18next** (v17) backed by **i18next** (v26) for internationalization.

### Key design choices:

| Concern | Choice | Rationale |
|---------|--------|-----------|
| Library | `react-i18next` | Mature, widely adopted, excellent React hook support |
| Namespace | Single `translation` namespace | App is small; single namespace avoids import complexity |
| Key format | 2-level dot notation (`namespace.key`) | Readable, avoids deep nesting |
| Locale files | `src/i18n/locales/en.json` + `de.json` | Co-located with source for easy contributor access |
| Storage key | `localStorage["oct-lang"]` | Simple, no server dependency |
| Detection order | localStorage → `navigator.language` prefix → fallback `"en"` | Respects explicit user choice, then browser preference |
| Fallback locale | `"en"` | English is the community-facing default |
| Language switch | `changeLanguage(locale)` from `src/i18n/index.ts` | Single entry point; updates i18next + localStorage atomically |
| LanguageSelector | Dropdown in Navigation bar | Persistent and accessible from all pages |

---

## Alternatives Considered

### 1. Manual React Context

A custom `LanguageContext` holding a locale string and a dictionary object. All components
would consume the context and look up strings by key.

**Rejected**: Requires significant boilerplate. No pluralization, interpolation, or fallback
support without re-implementing them. Hard to scale.

### 2. FormatJS / `react-intl`

The reference implementation for ICU message syntax. Excellent for complex pluralization and
date/number formatting.

**Rejected**: Heavier API surface, requires message extraction tooling, and ICU syntax is
harder for non-developer translators to work with. Overkill for a 2-locale app.

### 3. Lingui

A compile-time extraction approach with excellent TypeScript support.

**Rejected**: Requires a build step to extract and compile message catalogs. Adds CI
complexity and a steeper contributor onboarding curve. The runtime-only approach of i18next
is simpler for a small OSS project.

---

## Consequences

### Positive

- All user-visible strings externalized from JSX — zero raw string literals in components
- Language preference persists across sessions via localStorage
- Browser language auto-detected on first visit
- New translations can be contributed by editing a single JSON file
- Consistent fallback: missing German key → English value shown (no blank UI)

### Negative / Trade-offs

- Two new runtime dependencies (`i18next`, `react-i18next`)
- All new strings must be added to both `en.json` and `de.json` simultaneously
- Test setup requires `import "../src/i18n"` in `tests/setup.ts` so tests run in English

### Contributor Workflow

1. Add new string via `t("namespace.key")` in the component
2. Add English value to `src/i18n/locales/en.json`
3. Add German value to `src/i18n/locales/de.json`
4. Run `cd apps/frontend && npx vitest run tests/unit/i18n.test.ts` — the key-parity test
   will fail if either locale file is missing the new key

### Future Languages

To add a third language (e.g. French):

1. Create `src/i18n/locales/fr.json` mirroring `en.json`
2. Add `"fr"` to `SUPPORTED_LOCALES` in `src/i18n/index.ts`
3. Add `fr` config to `LOCALE_CONFIGS`
4. Open a GitHub issue using the `translation_contribution.yml` template
